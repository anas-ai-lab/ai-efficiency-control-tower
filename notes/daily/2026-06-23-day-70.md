## Comprehension Gate

**Kontext:** Drei Dokumentations-Artefakte die das System von
"funktioniert" zu "erklaerbar" machen. STRIDE klassifiziert
Bedrohungen als aktiv mitigiert (OK) oder kontextabhaengig (WARN/INFO).
README erklaert scope-genau was das System tut und nicht tut.
ADR-Review belegt dass alle Entscheidungen nachvollziehbar dokumentiert sind.

**Frage 1:** Was unterscheidet ein WARN-Punkt im Threat Model von
einem Fehler -- und warum waere es falsch, diese Punkte fuer das
aktuelle private Build als Fehler zu werten?

**Antwort:** Das sind Punkte die aktuell nicht relevant sind,
sondern erst bei Server-Deploy.

Claude ergaenzt: Genau -- der Kontext entscheidet. ChromaDB ohne
Auth ist kein Fehler wenn der Port nur localhost-intern erreichbar
ist und genau ein Nutzer das System bedient. Ein WARN sagt "das
wuerde sich aendern muessen, wenn der Deployment-Kontext sich
aendert" -- das ist engineering judgement, keine Luecke.

**Frage 2:** Warum ist der "Was AECT nicht ist"-Abschnitt in
der README besonders wichtig?

**Antwort:** So weiss man immer was man genau tun soll -- der
Scope/Rahmen wird gesetzt. Wenn jemand das nicht liest, kann er
sich ein falsches Bild vom Tool machen.

Claude ergaenzt: Konkret: Ein Interviewer der den Abschnitt
ueberspringt koennte fragen "Warum ersetzt das nicht den DSB?" und
erwartet eine Verteidigung statt zu verstehen dass das bewusst
ausserhalb des Scopes liegt. Der Abschnitt verhindert diese
Gespraeche, bevor sie entstehen.
