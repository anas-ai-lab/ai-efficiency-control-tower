# Day 28 — Idempotency-Keys für POST /triage

## Ziel
Idempotency-Schutz für POST /triage: wiederholte Requests mit demselben
`Idempotency-Key`-Header liefern das ursprüngliche Ergebnis zurück (200 +
`Idempotent-Replay: true`), statt den Use Case erneut zu verarbeiten.

## Gebaut
- `application/ports/idempotency_store.py` — neuer Port (Protocol),
  analog `RepositoryPort`. Kein Service-Eingriff.
- `adapters/in_memory/idempotency_store.py` — dict-Backend, Dev/Test.
- `adapters/sqlite/idempotency_store.py` — persistente Tabelle
  `idempotency_keys` in derselben DB-Datei wie `submitted_cases`.
- `adapters/api/dependencies.py` — `get_idempotency_store()`, gleiches
  Muster wie `get_triage_service()` (AECT_DB_PATH steuert In-Memory vs.
  SQLite).
- `adapters/api/routes/triage.py` — Header `Idempotency-Key` (optional,
  max. 200 Zeichen), Replay-Check vor `submit_use_case()`, Speichern danach.
- `app.py` — CORS `allow_headers` um `Idempotency-Key` erweitert.

## Architektur-Entscheidung
Idempotency-Check sitzt in der Adapter-Schicht, nicht im `TriageService`.
Begründung: Idempotency ist HTTP-/Transport-Anliegen, keine Domain-/
Anwendungslogik. `TriageService` unverändert — Dependency Inversion bleibt
intakt.

## Bewusste Grenze (ADR-005)
Kein Payload-Hash-Vergleich. Gleicher Key + geänderter Body → altes
Ergebnis, kein 409. Für privates Single-User-System ausreichend,
dokumentierte Limitation.

## Tests
- `tests/adapters/api/test_idempotency.py` — 3 Integrationstests
  (Replay-Match, unterschiedliche Keys, kein Header = unverändertes
  Verhalten).
- `tests/adapters/sqlite/test_idempotency_store.py` — 4 Unit-Tests
  (Persistenz, Überschreiben, unbekannter Key).
- Bugfix während der Umsetzung: Test-Fixture `_make_idempotency_app()`
  erzeugte Repository + Store pro Lambda-Call neu (kein geteilter State
  über Requests). Fix: Instanzen vor den Overrides erzeugen, per Closure
  referenzieren.
