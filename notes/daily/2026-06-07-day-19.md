  und der Package-Import zogen aus unterschiedlichen Verzeichnissen, weshalb
  sowohl `uv run pytest` als auch `uv run python -c "import aect"` scheiterten.

  kein dirty `git status` mehr nach Build-Operationen.

  war die ganze Zeit aktiv. Der echte Fehler war Umgebungskorruption.

### Block B — `roi_config.toml` + `calculate_roi()` Adapter
**Problem:** TOML-Keys (`PROFESSIONAL`, `LOW`, `HIGH`) stimmten nicht mit den
StrEnum-Values überein (`professional`, `pure_estimate`, `mandatory`).

**Lösung:**
- `roi_config.toml` vollständig auf StrEnum-Values umgestellt:
  - `EmployeeCategory`: junior / professional / senior / mixed (neu)
  - `EvidenceLevel`: pure_estimate / similar_project / tested_piloted
  - `AdoptionType`: mandatory / voluntary
- `calculate_roi()` implementiert — mappt `UseCaseInput` auf `_calculate_roi_values()`:
  - `frequency_per_year` ist bereits ein Jahreswert → `frequency_unit_value="ANNUALLY"`
    (Multiplikator 1, geht unverändert durch)
  - `country`-Parameter mit Default `"DE"`
- `tests/domain/test_roi_integration.py` (7 Tests): Enum-Exhaustion-Guards prüfen,
  dass alle StrEnum-Values in der TOML-Config vorhanden sind.

### Block C — `routing.py` — AI-vs-Automation-Router (Phase A)
Deterministisch, kein LLM-Call, seiteneffektfrei.

**4 Outcomes:**
- `AI_RECOMMENDED` — hohe Komplexität, unsichere Schätzung, ambigue Anforderung
- `AUTOMATION_RECOMMENDED` — einfach, hochfrequent, Pflichtnutzung
- `HUMAN_REVIEW_REQUIRED` — mind. 2 Datenschutz-/Risikoflags aktiv
- `BORDERLINE` — Mischsignale, Phase-C-LLM entscheidet später

**Architektur:** Signal-Sammlung (_collect_automation_signals, _collect_ai_signals,
_collect_risk_flags) strikt getrennt von Entscheidungslogik (_decide). Jede
Funktion einzeln testbar, alle pure functions.

**`requires_human_review` Property:** True wenn Empfehlung HUMAN_REVIEW_REQUIRED
OR mind. 1 Risikoflag aktiv — decoupled von der Routing-Empfehlung.
