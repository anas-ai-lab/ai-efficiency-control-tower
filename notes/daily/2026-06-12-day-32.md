# Day 32 — Injection Pattern Detection für sharpen_case (LLM01)

**Datum:** 2026-06-12
**Phase:** C

## Ziel
Defense-in-Depth gegen Prompt-Injection in `sharpen_case()`: Freitextfelder
werden vor dem LLM-Call auf bekannte Injection-Muster geprüft. Treffer werden
geloggt, der Call läuft trotzdem weiter (Flaggen, nicht Blocken).

## Was gebaut wurde
- `src/aect/application/sanitization.py`: `detect_injection_patterns()`,
  vier Regex-Pattern-Kategorien (ignore_instructions, role_hijack,
  delimiter_breakout, prompt_exfiltration), DE + EN.
- `tests/application/test_sanitization.py`: 13 Tests, inkl. False-Positive-
  Check gegen realistischen Geschäftstext.
- `service.py`: `sharpen_case()` prüft alle vier Freitextfelder vor dem
  LLM-Call. Bei Treffer: `logger.warning("injection_pattern_detected",
  case_id=..., fields=...)`. Kein Body/Prompt im Log (Allowlist v2.1).
- `tests/application/test_service.py`: zwei neue Tests
  (Injection-Treffer wird geloggt + blockiert nicht / sauberer Input loggt
  nichts).
- `tests/adapters/api/test_sharpen.py`: End-to-End-Red-Team-Test über die
  echte API — Injection-Payload führt weiterhin zu 200.

## Entscheidungen
- **Flaggen statt Blocken.** AECT ist Advisory-Layer, kein Gatekeeper. Ein
  hartes Blocken bei false positives würde legitime Einreichungen ablehnen
  (z. B. "ignoriere die alte Prozessbeschreibung" in echtem Business-Text).
- **Kein modul-globaler Logger.** `structlog.get_logger()` wird bei jedem
  Treffer neu erzeugt statt einmal beim Modul-Import — Grund: Caching-Effekt
  (`cache_logger_on_first_use=True`) hätte den Logger an die zum Zeitpunkt
  des ersten Aufrufs aktive Prozessor-Kette gebunden und `capture_logs()`
  in Tests ignoriert, wenn vorher schon ein anderer Test denselben Logger
  "verbraucht" hatte.
- Logging-Allowlist v2.1 wurde faktisch um ein Feld erweitert (`fields`:
  Feldname → Pattern-Namen). Enthält keine PII/Body/Prompt — als bewusste
  Erweiterung dokumentiert, Checkliste wird in Phase E/F formal nachgezogen.

## Stolperstein
Erster Versuch des Service-Patches hatte drei Probleme gleichzeitig:
Import-Reihenfolge (`from __future__` nicht an erster Stelle), neuer Code
landete versehentlich *innerhalb* von `if case is None:` (hätte bei Treffer
`None` zurückgegeben statt Flaggen-Verhalten), und Logger-Caching brach
`capture_logs()` im Suite-Lauf. Datei komplett neu geschrieben statt
Patch-für-Patch — bei verschachtelten Strukturfehlern schneller.
