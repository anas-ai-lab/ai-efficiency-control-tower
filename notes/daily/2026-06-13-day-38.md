# Tag 38 ÔÇö Function-Calling-Loop verdrahtet (Phase C)

**Datum:** 2026-06-13
**Phase:** C (LLM-Integration, Mock-First)

## Was wurde gebaut

- ADR-0009: Loesung des offenen Punkts aus ADR-0008 -- Function-Calling-
  Loop lebt inline in `TriageService.propose_solution()`, kein eigener
  `ToolCallingLLMAdapter`. Max. zwei `complete()`-Aufrufe pro Call, kein
  while-Loop (LLM10 Unbounded Consumption).
- `prompts/propose_solution/v2/` (system.md + user.md): System-Prompt
  weist auf `lookup_stack_options` hin und markiert Plattform-
  Beschreibungen als vorlaeufig/unbelegt (RAG folgt Phase D). v1 bleibt
  unveraendert erhalten (Versionierung).
- `propose_solution()`: Default-Prompt-Version `"v1"` -> `"v2"`. Erster
  `complete()`-Call bietet `TOOL_DEFINITIONS` an. Bei `tool_calls`:
  Assistant-Nachricht mit `tool_calls` anhaengen, jeden Call via
  `dispatch_tool_call()` ausfuehren, Ergebnis als `role="tool"`-Nachricht
  (JSON, `tool_call_id`) anhaengen, zweiter `complete()`-Call -> finale
  Antwort. Beide Calls werden via `log_llm_cost()` geloggt.
- LLM06 (Excessive Agency): `UnknownToolError` wird abgefangen und als
  `{"error": ...}`-Tool-Ergebnis an das LLM zurueckgegeben statt die
  Anfrage abzubrechen -- Graceful Degradation.
- `ports/llm.py`: ToolCall-Docstring aktualisiert (Tag-38-Verweis aufgeloest,
  Begruendung warum `arguments`-Validierung heute nicht noetig ist).

## Tests

- 4 neue Tests in `tests/application/test_service.py`:
  `TestTriageServiceProposeSolution` (3) + `TestTriageServiceProposeSolutionUnknownTool` (1).
- Neue Fake-Klasse `_UnknownToolLLMAdapter` (simuliert Tool-Anfrage fuer
  nicht registriertes Tool).
- `tests/adapters/api/test_propose_solution.py`: Assertion `prompt_version
  == "v1"` -> `"v2"` (Folge der Default-Aenderung).
- `uv run pytest -q`: 274 passed (270 + 4), Coverage weiterhin >= 99%.

## Was bewusst NICHT gemacht wurde

- Kein `while`-Loop fuer mehrfache Tool-Call-Runden (ADR-0009,
  dokumentierte Grenze).
- Keine Validierung von `tool_call.arguments` gegen ein Schema --
  `lookup_stack_options` ist parameterlos, kein aktueller Bedarf.
- Keine RAG-Anbindung fuer `lookup_stack_options` -- weiterhin Phase D.
- `sharpen_case()` unveraendert (kein `tools`-Parameter).

## Offene Punkte

- Phase-D-Vorab-Check: EU-AI-Act-Stand re-verifizieren (Digital-Omnibus-
  Amtsblatt-Status, session-protocol v3 ┬º4), bevor Compliance-Wissensbasis
  geschrieben wird.
- ADR-0009 dokumentierte Grenze (max. 2 complete()-Calls) erweitern, sobald
  ein zweiter Tool-Bedarf oder echtes Mehrfach-Tool-Call-Verhalten eines
  realen Providers beobachtet wird.

## Comprehension Gate -- Antworten

1. Maximal zwei "Hin und Her"-Schritte (zwei `complete()`-Aufrufe) --
   unabhaengig davon, ob ueberhaupt ein Tool angefragt wird.
2. Der Code bricht nicht ab. Der Assistent bekommt eine Fehlermeldung als
   Werkzeug-Ergebnis zurueck und formuliert trotzdem eine finale Antwort
   (Graceful Degradation, LLM06).
